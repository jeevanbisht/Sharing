New-EntraBetaPrivateAccessApplicationSegment – Bulk Creation Guide

Walkthrough of the CSV structure, PowerShell script behavior, and common gotchas when using New-EntraBetaPrivateAccessApplicationSegment for bulk segment creation.

1. CSV structure

AppSegments.csv format

Your AppSegments.csv should look like this:

csvAppObjectId,DestHost,ports,protocol,type
00001111-aaaa-2222-bbbb-3333cccc4444,10.106.97.0/24,"1-21,23-442,444-65535","TCP,udp",ipRangeCidr
00001111-aaaa-2222-bbbb-3333cccc4444,10.106.96.0/24,"1-21,23-442,444-65535","udp",ipRangeCidr
00001111-aaaa-2222-bbbb-3333cccc4444,10.106.95.0/24,"1-21","udp",ipRangeCidr

Header row must be on its own line, followed by one line per entry (as shown above).

ports is a comma-separated list of ranges, for example:
1-21,23-442,444-65535

protocol is a comma-separated list, e.g. "TCP,udp" or "udp". Casing doesn’t matter; the cmdlet normalizes to lowercase.

type is ipRangeCidr here, which matches the DestinationType enum for CIDR ranges.

2. Script behavior

CreateAppSegments.ps1

Your CreateAppSegments.ps1 script:

powershell$csvFile = "C:\temp\AppSegments.csv"

# Assuming the CSV file has columns named 'AppObjectId', 'DestHost', 'ports', 'protocol', 'type'
$variables = Import-Csv $csvFile

foreach ($variable in $variables) {
    $appObjectId = $variable.AppObjectId
    $destHost    = $variable.DestHost
    $ports       = $variable.ports -split ","
    $protocol    = $variable.protocol -split ","
    $type        = $variable.type

    $params = @{
        ApplicationId   = $appObjectId
        DestinationHost = $destHost
        Ports           = $ports
        Protocol        = $protocol
        DestinationType = $type
    }

    New-EntraBetaPrivateAccessApplicationSegment @params
}

What this does:

  • Imports each CSV row into $variables as an object.
  • For each row:
    • Splits ports into an array of ranges (e.g. @("1-21","23-442","444-65535")).
    • Splits protocol into an array (e.g. @("TCP","udp")).
    • Builds a hashtable $params and splats it into New-EntraBetaPrivateAccessApplicationSegment.

This matches Example 3 from the official cmdlet docs. The parameters -Ports and -Protocol accept string or string[], and passing arrays is fully supported.

3. Execution flow

How to run it end-to-end

Run the following in an elevated PowerShell session:

  1. Install / update the Entra Beta module 
    powershellInstall-Module Microsoft.Entra.Beta -Force
Import-Module Microsoft.Entra.Beta
  2. Connect with the right scopes 
    powershellConnect-Entra -Scopes `
    'NetworkAccessPolicy.ReadWrite.All', `
    'Application.ReadWrite.All', `
    'NetworkAccess.ReadWrite.All'
  3. Confirm your app object ID 
    powershellGet-EntraBetaApplication -Filter "displayName eq '<GlobalSecureAccess_Application_DisplayName>'" |
    Select-Object DisplayName, Id

    Make sure the Id matches 00001111-aaaa-2222-bbbb-3333cccc4444 in your CSV.

  4. Save files and run
    • Save the CSV as C:\temp\AppSegments.csv
    • Save the script as C:\temp\CreateAppSegments.ps1
    powershellSet-Location C:\temp
.\CreateAppSegments.ps1
4. Known gotcha

Older module bug

Older versions of Microsoft.Entra.Beta.NetworkAccess can cause New-EntraBetaPrivateAccessApplicationSegment to throw a CmdletBinding error (for example, PropertyNotFoundForType around ParameterSetName).

Mitigation Update to a supported module version

If you see an error like this:

  • Ensure you’re on a recent Microsoft.Entra.Beta (e.g. 1.1.x or later).
  • If multiple versions are installed:
    • Uninstall the older version.
    • Re-import the latest:
powershellGet-Module Microsoft.Entra.Beta -ListAvailable
Uninstall-Module Microsoft.Entra.Beta -RequiredVersion <oldVersion>
Import-Module Microsoft.Entra.Beta -RequiredVersion <newVersion>
